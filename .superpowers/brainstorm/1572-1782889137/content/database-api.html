数据库模型 & API 设计

核心要点：模拟数据与真实硬件共用同一套数据表和API接口

🗄️ ER 实体关系图

MySQL 数据库核心表结构 (8张表)
📋 plot (地块表)
id, name, location, area, crop_type,
owner_id(FK→user), create_time
📋 device (设备表)
id, device_code, device_type(温湿度/土壤/灌溉),
plot_id(FK), status(在线/离线/故障),
mqtt_topic, last_heartbeat
📋 sensor_data (传感器数据表)
id, device_id(FK), data_type, data_value,
unit, collect_time, source(模拟/真实)
⭐ 模拟/真实数据写入同一张表
📋 alert_rule (告警规则表)
id, device_id(FK), metric_type,
min_threshold, max_threshold,
enabled, notify_method
📋 alert_log (告警记录表)
id, rule_id(FK), device_id(FK),
alert_msg, alert_level, current_value,
is_read, trigger_time
📋 control_log (控制指令表)
id, device_id(FK), command(开/关),
operator_id(FK), command_status,
result_msg, send_time, exec_time
⭐ MQTT下发/模拟执行共用
📋 user (用户表)
id, username, password(BCrypt),
role(农户/农场管理员/系统管理员),
phone, create_time
📋 knowledge_base (知识库表)
id, question, answer, category,
keywords, create_time
⭐ AI智能体RAG检索用

🔌 核心 REST API (共 20+ 接口)

API 按模块分组 — 前缀 /api/v1
数据监测模块
方法路径说明
GET/sensor/realtime/{deviceId}获取设备最新数据
GET/sensor/history?deviceId=&days=7历史数据趋势（7天）
GET/plot/{id}/overview地块综合概览
GET/plot/list多地块列表
设备控制模块
方法路径说明
POST/control/irrigation灌溉开关控制
GET/control/log?deviceId=控制历史记录
GET/device/status/{id}设备在线状态
POST/device/bind绑定/解绑设备
告警管理 & AI问答
方法路径说明
POST/alert/rule设置告警阈值
GET/alert/log?page=&size=告警日志分页
POST/ai/chatAI智能问答
GET/ai/knowledge知识库管理
🔑 关键设计决策：模拟/真实切换
sensor_data 表有 source 字段区分来源。后端通过 DataCollector 接口抽象，有 MockDataCollector 和 MqttDataCollector 两个实现， 通过配置文件切换。API接口完全不变，上层代码零改动。
