## 1. 项目背景与目标

### 1.1 项目背景

随着农业现代化与物联网技术的发展，传统农业正逐步向数字化、智能化转型。实时监测农田环境参数（土壤湿度、温度、光照、CO₂浓度等），并结合人工智能分析实现精准管理，已成为提高产量、节约资源的重要手段。

本次实训项目依托中软国际5R体系，以"端-智-云一体化"为设计理念，分两阶段实施：校内阶段基于模拟数据发生器完成软件主线（数据传输、智能体构建、可视化大屏），基地阶段接入真实鸿蒙开发板与传感器，实现全链路联调。本项目要求构建一套智慧农业监测系统，满足数据采集、智能控制、可视化分析和智能问答等核心需求。

### 1.2 项目目标

- 实时采集并展示土壤湿度、温度、光照等环境参数，同时呈现设备在线状态
- 支持远程手动/自动控制灌溉设备，具备阈值告警与联动功能
- 提供历史数据趋势分析（折线图，自动降采样）、多地块总览大屏
- 构建基于知识库的智能问答体（RAG），为农户提供灌溉建议与农事指导
- 实现设备绑定管理、告警日志查询、用户RBAC权限控制等系统管理功能
- 具备模拟数据与真实硬件数据**无缝切换**的能力（DataCollector接口抽象），保证代码可复用性与可扩展性

## 3. 用户角色与权限（RBAC三级模型）

| 角色 | 系统标识 | 权限描述 |
|------|---------|---------|
| **农户** | FARMER | 注册/登录，查看所辖地块的实时数据（土壤湿度、温度、光照等），查看历史趋势图（自动降采样），接收告警通知（WebSocket实时推送），远程控制灌溉设备，设置个人阈值，使用智能问答（AI注入实时环境参数），查看设备在线状态。**SQL级数据隔离**：所有查询强制附加 `WHERE owner_id = currentUserId` |
| **农场管理员** | ADMIN | 全局视图：查看所有地块的数据总览，管理地块CRUD，管理设备绑定/解绑（含仓库中未绑定设备），配置全局阈值，查看告警日志，管理农户账号。**跳过owner过滤**：可查看全部数据 |
| **系统管理员** | SUPER_ADMIN | 系统运维管理：用户管理（增删改查/重置密码/启用禁用），设备注册与解绑，系统参数配置（采集频率、MQTT地址），查看所有操作日志及审计追踪 |

**RBAC实现方式**：`RbacUtils.getOwnerFilterId()` — ADMIN返回null（不过滤），FARMER返回currentUserId（行级过滤）。

## 4. 业务流程描述

### 4.1 数据监测与告警流程

1. 农户在平台设置土壤湿度阈值（或使用系统默认值）
2. **校内阶段**：MockDataCollector动态轮询在线SENSOR设备，生成含昼夜节律的模拟数据 → 直接写入`sensor_data`表 → WebSocket推送
3. **基地阶段**：硬件传感器采集数据 → MQTT上报 → MqttDataCollector接收 → 写入`sensor_data`表 → WebSocket推送
4. 前后端通过`DataCollector`接口抽象切换，**业务代码零改动**
5. AlertEngine实时判断阈值，超限触发告警 → 写入`alert_log`表 → WebSocket实时推送前端
6. HeartbeatChecker每60秒检测心跳，3分钟无心跳自动标记OFFLINE → 告警+UI置灰

### 4.2 灌溉控制流程（手动/自动）

**手动模式**：
1. 农户选择地块 → 选择CONTROLLER设备 → 点击开启/关闭灌溉
2. 后端执行**四级校验链**：①归属权校验 ②设备类型校验（必须CONTROLLER） ③在线状态校验 ④审计日志写入（REQUIRES_NEW独立事务）
3. MQTT下发指令 → **10秒超时闭环等待** → SUCCESS/TIMEOUT/FAILED → WebSocket推送结果
4. 所有操作（含失败和超时）强制写入`control_log`表，永不丢失

**自动模式（创新功能）**：
1. 当土壤湿度低于阈值且持续一定时间（可配置），AlertEngine自动触发灌溉指令
2. 湿度恢复至正常范围后自动关闭
3. 自动操作也在`control_log`表中记录，`source=AUTO`

### 4.3 智能问答咨询流程

1. 农户打开AI对话界面，输入问题（如"当前土壤湿度合适吗？需要浇水吗？"）
2. 后端RagService处理：
   - 校验地块归属
   - 查询该地块所有SENSOR最新数据 → **组装实时环境上下文**
   - RAG检索`knowledge_base`表（MySQL全文索引 + 关键词匹配）
   - 组装Prompt：`[环境参数: 温度26.5°C, 湿度68%...] + [知识库参考] + 用户问题`
   - 规则引擎匹配回答 → 未匹配则降级LLM API
3. 返回针对性诊断建议（非通用回答）

### 4.4 设备绑定管理流程

1. 系统管理员在后台查看仓库中未绑定设备（`plot_id IS NULL`）
2. 选择目标地块，执行绑定 → 校验设备未被占用 → 更新`plot_id`
3. 设备上线后发送心跳，`last_heartbeat`更新，HeartbeatChecker监控
4. 管理员可解绑设备 → `plot_id=NULL`，设备回到仓库
5. **MockDataCollector动态发现**：新绑定设备在下一个采集周期（5秒内）自动开始生成数据，无需重启

### 4.5 数据可视化与报表导出

1. 历史数据趋势折线图（7天/30天/自定义），**自动降采样**确保≤500数据点
2. 降采样策略：≤24h→5分钟粒度, ≤7天→1小时, ≤30天→4小时, >30天→1天
3. 多地块总览仪表盘（Edge Fluent卡片式布局）
4. 支持导出历史数据报表（Excel格式）

## 7. 数据模型（8张核心表）

> 完整DDL见 `smart-agriculture-server/src/main/resources/db/schema.sql`  
> 预置测试数据见 `smart-agriculture-server/src/main/resources/db/data.sql`

### 7.1 表结构总览

| 表名 | 对应原文档 | 说明 | 关键字段 |
|------|-----------|------|---------|
| `user` | user | 用户表 | id, username, password(BCrypt), role(FARMER/ADMIN/SUPER_ADMIN), phone, enabled |
| `plot` | farm | 地块表 | id, name, location, area, crop_type, **owner_id**(FK→user) |
| `device` | device | 设备表 | id, device_code, device_name, device_type, **device_category(SENSOR/CONTROLLER)**, plot_id, status, protocol, last_heartbeat |
| `sensor_data` | sensor_data | 传感器数据（EAV模式） | id, device_id, data_type, data_value, unit, collect_time, source(MOCK/MQTT) |
| `alert_rule` | threshold_config | 告警规则表 | id, device_id, plot_id, metric_type, min_threshold, max_threshold, alert_level(CRITICAL/WARNING/INFO) |
| `alert_log` | alarm_record | 告警记录表 | id, rule_id, device_id, plot_id, alert_msg, alert_level, current_value, is_read, trigger_time |
| `control_log` | control_log | 控制指令日志 | id, device_id, operator_id, command, command_status(PENDING→SENT→SUCCESS/FAILED/TIMEOUT), result_msg, send_time, response_time |
| `knowledge_base` | knowledge_doc | 知识库表 | id, question, answer, category, keywords, crop_type |

### 7.2 与原文档的核心差异说明

| 差异点 | 原文档 | 当前实现 | 原因 |
|--------|--------|---------|------|
| 传感器数据模型 | 宽表（humidity/temperature/light/co2列） | **EAV模型**（data_type + data_value） | 支持动态扩展传感器类型，无需ALTER TABLE |
| 设备分类 | 单一type字段 | **device_category**（SENSOR/CONTROLLER） | 区分"采集型"和"控制型"设备，支撑四级校验链 |
| 用户角色 | 农户/农场管理员/系统管理员 | FARMER/ADMIN/SUPER_ADMIN + RBAC工具类 | 编程化角色判断，SQL级数据隔离 |
| 数据采集 | 直接MQTT连接 | **DataCollector接口抽象** + MockDataCollector | 模拟/真实无缝切换，业务代码零改动 |
| 告警规则 | threshold_config表 | alert_rule表 + AlertEngine动态扫描 | 支持多级告警、按设备/地块粒度配置 |
| 控制指令 | 简单下发 | **四级校验链 + 10秒超时闭环 + REQUIRES_NEW审计** | 防呆防伪 + 超时兜底 + 审计合规 |
| 知识库 | 含embedding_vector向量 | MySQL全文索引 + 关键词匹配 | 校内阶段无需向量数据库，可后续扩展 |

## 9. 项目创新点

> 以下创新点在原始需求分析文档基础上进行了架构级增强，使项目从"课程作业"级别提升到"企业级可部署"标准。

### 9.1 架构级创新（4项）

| # | 创新点 | 原始方案 | 创新实现 | 价值 |
|---|--------|---------|---------|------|
| 1 | **DataCollector接口抽象** | 直接MQTT连接 | 接口 + MockDataCollector/MqttDataCollector双实现 + application.yml配置切换 | 校内15天代码直接用于基地10天，零重构 |
| 2 | **设备职能分类** | 单一device_type | device_category(SENSOR/CONTROLLER) + 前端v-if动态渲染 + 空状态组件 | 设备有明确职能分工，防止传感器被误控 |
| 3 | **四级指令校验链** | 直接下发指令 | 归属权→设备类型→在线状态→审计日志 | 防呆防伪防越权，生产级指令管道 |
| 4 | **审计日志事务隔离** | 普通事务 | ControlLogService REQUIRES_NEW独立事务 | 业务异常不回滚审计记录，合规底线 |

### 9.2 防御级创新（4项）

| # | 创新点 | 原始方案 | 创新实现 | 价值 |
|---|--------|---------|---------|------|
| 5 | **RBAC动态数据权限** | 简单角色判断 | RbacUtils: ADMIN返回null(全局), FARMER返回userId(行级过滤) | SQL级数据隔离，水平越权防护 |
| 6 | **历史数据降采样** | 全量返回 | SensorService四档TimeGranularity(5min/1h/4h/1d) + 上限500点 | 防OOM + 前端流畅渲染 |
| 7 | **心跳离线自检** | 人工查看状态 | HeartbeatChecker每60秒扫描 + 3分钟超时自动OFFLINE | 自动化运维，无需人工巡检 |
| 8 | **@Scheduled线程池隔离** | 默认单线程 | SchedulingConfig(poolSize=10) | 数据采集/心跳检测/告警扫描互不阻塞 |

### 9.3 体验级创新（4项）

| # | 创新点 | 原始方案 | 创新实现 | 价值 |
|---|--------|---------|---------|------|
| 9 | **模拟数据昼夜节律** | 完全随机数据 | 正弦节律(峰值14:00) + 高斯噪声(σ=0.3°C) + 慢速趋势 + 2%降雨事件 | 路演时ECharts曲线平滑自然 |
| 10 | **AI上下文注入** | 通用知识库问答 | 实时sensor_data注入LLM Prompt，结合当前地块环境参数 | 回答针对性强，非通用模板 |
| 11 | **指令超时闭环** | 无超时机制 | CompletableFuture 10s等待 + WebSocket实时推送结果 | 用户知道指令执行状态 |
| 12 | **动态轮询+批量写入** | 硬编码设备ID + 单条insert | 每次查DB获取活跃SENSOR + saveBatch批量写入 | 新设备无需重启即可用 |

### 9.4 预置数据场景设计

| 场景 | 配置 | 验证目标 |
|------|------|---------|
| 地块A（温室大棚） | SENSOR + CONTROLLER | 灌溉按钮正常显示与操作 |
| 地块B（露天农田） | 仅SENSOR，无CONTROLLER | **空状态降级**："当前农田未配置控制设备" |
| 仓库中 | 1台未绑定CONTROLLER | 管理员可分配绑定 |
| 农户隔离 | farmer1仅能访问地块A | SQL级越权拦截验证 |

## 10. 架构实现亮点总结

### 10.1 模拟/真实硬件无缝切换

```
校内15天: MockDataCollector (动态轮询 + 正弦节律)
基地10天: MqttDataCollector (订阅MQTT Topic)
          ↓
    DataCollector 接口
          ↓
    application.yml: data.collector = mock | mqtt
          ↓
    业务代码 / API / 数据库 / 前端 —— 完全不变
```

### 10.2 控制指令全链路审计

```
前端按钮 → Controller → ControlService（四级校验）
  ├─ 归属权 → 设备类型 → 在线状态
  └─ ControlLogService.insertSentLog() [REQUIRES_NEW → 已提交]
       └─ MQTT下发 → 10s CompletableFuture
            ├─ SUCCESS → markSuccess() [REQUIRES_NEW]
            ├─ TIMEOUT → markTimeout() [REQUIRES_NEW]
            └─ FAILED  → markFailed()  [REQUIRES_NEW]
                           ↓
            审计日志永不丢失（即使异常回滚也不影响）
```

### 10.3 前端双端差异化设计

| 维度 | PC端（管理员） | 移动端（农户） |
|------|--------------|--------------|
| 框架 | Vue 3 + Element Plus | uni-app |
| 风格 | Edge Fluent卡片式Dashboard Hub | 底部Tab导航 |
| 核心功能 | 数据大屏、多地块总览、设备管理、Excel导出 | 实时监测、灌溉控制、AI问答、告警接收 |
| 空状态 | el-empty组件 | uni-app空状态模板 |
| 设备渲染 | device_category驱动 | 同 |
